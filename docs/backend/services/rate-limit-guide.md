# Groq API Rate Limit 完整指南

本文檔說明 Groq API 的速率限制、當前配置、優化建議和故障排除方法。

---

## 📊 免費額度限制

### 模型限制

| 模型                    | 用途 | RPM | RPD    | TPM    | TPD     |
| ----------------------- | ---- | --- | ------ | ------ | ------- |
| Llama 3.1 8B Instant    | 評估 | 30  | 14,400 | 14,400 | 500,000 |
| Llama 3.3 70B Versatile | 摘要 | 30  | 14,400 | 6,000  | 100,000 |

### ⚠️ 重要說明

**Rate limit 是組織層級的，所有模型共用同一個限制！**

---

## 🚦 Rate Limit 機制

### HTTP 429 錯誤

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 639
```

Retry-After 是 Groq API 動態計算的，基於你的累積使用量。

### 滾動窗口

Groq 使用滾動窗口追蹤請求，需要等待最早的請求"過期"才能繼續。

## ⚙️ 當前配置

```python
# app/services/llm_service.py
CONCURRENT_LIMIT = 2      # 並發數
REQUEST_DELAY = 4         # 延遲（秒）
MAX_RETRIES = 2           # 最大重試
API_TIMEOUT = 30          # 超時（秒）
```

**速率計算：** 2 × (60/4) = 30 RPM ✅

## 📈 使用量估算

### /news_now 指令

- 100 篇文章 = 200 API 調用 = ~7 分鐘
- 50 篇文章 = 100 API 調用 = ~3.5 分鐘
- 每日可執行：14,400 ÷ 200 = 72 次

### 背景排程器

- 每 6 小時處理 50 篇 = 100 API
- 每天 4 次 = 400 API
- 遠低於 14,400 限制 ✅

## 🐛 故障排除

### 頻繁遇到 429

- 增加延遲到 5-6 秒
- 降低並發數到 1
- 增加測試間隔到 15 分鐘

### 等待時間過長

- 使用背景排程器
- 升級付費方案
- 接受較長處理時間

---

## 🎯 最佳實踐

1. **預估使用量** - 確保不超過配額
2. **優雅降級** - 記錄失敗文章，稍後重試
3. **使用者通知** - 顯示預估時間
4. **背景處理** - 使用排程器
5. **快取結果** - 避免重複處理
